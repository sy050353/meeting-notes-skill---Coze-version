# 会议纪要规则体系

> **⚠️ 本文件是会议纪要规则的唯一可信源。** 所有相关规则以此文件为准，MEMORY.md 和 USER.md 仅作引用和摘要，不维护完整规则。
>
> **自主维护要求：** 每次处理会议纪要相关任务时，必须先读取本文件。涉及规则变更时立即更新本文件，不等用户提醒。

## 🚨 最高优先级：发送与状态检查清单

### 场景1：用户请求发送待发送纪要时
**触发条件：** 用户明确请求「发送纪要」「推送纪要」「发送待发送的」

**执行顺序（严格按步骤）：**

| 步骤 | 动作 | 要求 |
|------|------|------|
| 1️⃣ | **展示纪要内容** | 逐条完整展示所有 `[待发送]` 的纪要内容 |
| 2️⃣ | **更新纪要状态** | 将所有已发送的纪要从 `[待发送]` 改为 `[已发送]` |
| 3️⃣ | **数量校验** | 核对：发现X条 → 实际发送X条 → 状态更新X条，三者必须一致 |

**⚠️ 强制顺序：先展示内容，再更新状态。绝对不能先更新状态再展示。**

**不同情况的处理：**
- **有待发送纪要** → 逐条完整展示纪要内容
- **无待发送纪要** → 回复「当前无待发送的会议纪要」

**禁止行为：**
- ❌ 只回复状态提示（如「已发送」）而不展示纪要内容
- ❌ 静默不回复
- ❌ 先更新状态再展示内容

---

### 场景2：收到用户会议内容时
**触发条件：** 用户提供会议内容（含产品名、参会方、要点等）或明确说「记一下会议」

**执行顺序（严格按步骤）：**

| 步骤 | 动作 | 要求 |
|------|------|------|
| 1️⃣ | **读取本文件完整规则** | 不能只靠记忆，必须对照检查 |
| 2️⃣ | **生成英文草稿** | 不写入文件，只作为预览 |
| 3️⃣ | **质量自检** | 按「质量自检清单」逐项检查 |
| 4️⃣ | **预览给用户确认** | 清晰展示完整草稿 |
| 5️⃣ | **用户确认后写入** | 用户明确说「确认」「没问题」「写入」等才标记 [待发送] 并写入文件 |

**确认标准：**
- ✅ 有效确认：「确认」、「好的，写入」、「没问题」、「可以」
- ❌ 不算确认：用户没回复、用户提出修改意见、用户说「等下」

**多段内容处理：**
- 用户分多次发送会议内容时，等用户说「发完了」「以上是全部」后再生成草稿
- 不要每发一段就生成一版

---

### 场景3：发送后状态更新
**触发条件：** 已展示纪要内容后

**执行动作：**
1. 打开对应日期的纪要文件
2. 将刚发送的纪要标记从 `## [待发送]` 改为 `## [已发送]`
3. 核对数量：发送了几条，就改几条的状态
4. 确保没有遗漏（包括之前积压的待发送记录）

**禁止：** 发送后不更新状态，导致下一次重复发送。

## 📝 一、会议记录模板与格式规范

### 1.1 标准模板

```
[产品名/主题大写] | YOYO [月份缩写 日期] | w/ [角色/部门名]
- [完整句子表达的要点1]
- [完整句子表达的要点2]
- [完整句子表达的要点3]

F/O: [具体跟进事项 或 good for now]
```

### 1.2 格式要点

| 项目 | 规范 | 示例 |
|------|------|------|
| **标题格式** | 产品名/主题 \| YOYO 月份缩写+日期 \| w/ 角色/部门 | `RFQE | YOYO Jun 02 | w/ PM` |
| **产品名** | 全部大写 | `EMSX API`、`FIXNET`、`RBLD` |
| **日期格式** | 月份缩写 + 空格 + 日期（两位数），不加年份 | `Jun 02`、`Apr 15` |
| **语言** | 全部英文 |  |
| **要点** | 完整句子，不是词组；2-5个要点为宜 | `- Discussed testing progress` |
| **F/O 位置** | 另起一行，顶格，前面空一行 |  |
| **F/O 内容** | 有明确跟进事项写具体内容，没有统一写 `good for now` | `F/O: keep in touch`、`F/O: good for now` |

### 1.3 F/O 规则（重要）

**什么时候写具体 F/O：**
- 用户明确提到「跟进」、「follow up」、「下一步」、「待办」等触发词
- 会议中有明确的行动项和责任人

**什么时候写 `good for now`：**
- 没有明确的跟进事项
- 用户没有提到待办或下一步
- 会议只是信息同步，没有行动项

**⚠️ 禁止：** 自行推断或添加待办事项。没有明确触发词时，统一写 `good for now`。

### 1.4 不同会议长度的要点数量

| 会议类型 | 要点数量 | 说明 |
|---------|---------|------|
| 短沟通（call/email/IM） | 2-3 行 | 简洁明了 |
| 常规会议（30-60分钟） | 3-5 个要点 | 提炼核心内容 |
| 长会议（60分钟以上） | 5-7 个要点 | 保留关键信息，去除冗余 |

**原则：** 用完整句子，相关联的要点可以合并成一句话，但不能只是名词堆砌。

## 📤 三、发送与推送规则

### 3.1 发送原则

**不遗漏 > 按需发送**

每次执行发送任务时：
1. 先检查所有「待发送」和「发送失败」的历史纪要
2. 有就先补发，再处理当前时段新记录
3. 逐条完整展示纪要内容

### 3.2 发送方式

**唯一正确方式：逐条完整展示纪要内容**

- 在对话中逐条回复完整纪要内容
- 展示完成后立即更新状态为 `[已发送]`

### 3.3 状态标记

| 标记 | 含义 | 说明 |
|------|------|------|
| `## [待发送]` | 等待发送 | 用户确认后写入，等待发送 |
| `## [已发送]` | 已发送成功 | 发送后立即更新 |
| `## [发送失败]` | 发送失败 | 需重试或人工处理 |

### 3.4 发送失败处理

如果发送遇到问题：
1. 标记状态为「发送失败」
2. 告知用户发送失败
3. 等待用户指示重试或人工处理

## ✅ 四、质量自检清单

每次生成纪要草稿后，必须逐项核对：

- [ ] **标题格式**：产品名大写 | YOYO Mon DD | w/ 角色
- [ ] **日期格式**：月份缩写 + 日期（如 Jun 02），不加年份
- [ ] **全英文**：没有中文内容（专有名词除外）
- [ ] **完整句子**：要点是完整句子，不是名词词组
- [ ] **术语正确**：产品名、机构名、术语与术语表一致
- [ ] **F/O 格式**：另起一行，顶格，前面有空行
- [ ] **F/O 内容**：有明确触发词写具体事项，否则写 good for now
- [ ] **无 emoji**：整个纪要不包含表情符号
- [ ] **无具体时间**：只保留日期，不标注上午/下午/几点
- [ ] **视角正确**：从 Yoyo 的角度撰写，对方观点用 noted 等表述

## 📂 五、文件结构与命名规范

### 5.1 目录结构

```
会议纪要/
├── 2026-04-15.md         # 每日纪要文件
├── 2026-04-16.md
├── ...
└── 索引/                 # 月度索引目录（自动生成）
    ├── README.md         # 总索引（按月份跳转）
    ├── 2026-04.md        # 4月纪要索引（按日期/产品/参会方）
    ├── 2026-05.md
    └── ...
```

### 5.2 每日文件格式

```markdown
## [状态1]

**标题1**
- 要点1
- 要点2

F/O: xxx

## [状态2]

**标题2**
- 要点1
- 要点2

F/O: xxx
```

- 每条纪要以 `## [状态]` 开头
- 状态与正文之间空一行
- 多条纪要之间空两行

### 5.3 索引说明

**索引文件位置：** `会议纪要/索引/`

**索引类型：**
- **总索引（README.md）**：按月份汇总，快速跳转到各月索引
- **月度索引（YYYY-MM.md）**：包含三种维度
  - 按日期：查看某一天的所有会议
  - 按产品/主题：查看某产品相关的所有会议
  - 按参会方：查看与某类联系人的所有会议

**索引更新：**
- 每次新增/修改纪要文件后，运行 `generate_index.py --meeting-dir <纪要目录>` 重新生成索引
- 状态标记：✅ 已发送 | ⏳ 待发送 | ❌ 发送失败
